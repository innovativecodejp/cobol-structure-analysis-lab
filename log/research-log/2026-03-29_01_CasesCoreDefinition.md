# 2026-03-29_01_CasesCoreDefinition

## 🎯 今日の研究焦点（1つだけ）
- `70_cases` 全体の基礎として、`Case` を「事例」ではなく **構造的判断単位** として定義し、`Scope / Guarantee / Decision` との非同一性を固定する。

## 🏗 モデル仮説
- `Case` は、理論を現実の判断材料へ接続する **検証層** である。
- `Case` は AST ノード、Guarantee Unit、Decision 結論とは異なる。
- `Case` は、構文・構造・判断の三層を束ねるが、それらを混同しない記述単位である。
- `Case` が曖昧だと、後続の分類・比較・検証が恣意的になる。

## 🔬 構造設計（触った層：AST/CFG/DFG）
- **AST-facing**：Case は AST へトレース可能であるが、AST 部分木そのものではないと整理した。
- **CFG-facing**：Case は制御構造の複雑性を観測する単位として位置づけた。
- **DFG-facing**：Case はデータ依存と境界横断を説明できる単位として位置づけた。
- **Judgment-facing**：Guarantee 候補と Decision 論点を同じ記述に載せる枠として整理した。

## ✅ 今日の決定事項
- `Case` を **構造的判断単位** と定義した。
- `Case` の最小構成要素として、識別子・対象範囲・構造要約・保証候補・判断論点などを整理した。
- `Case` と `Scope / Guarantee / Decision` の違いを個別に分けて記述した。
- `Case` の抽象度を、構文層・構造層・判断層の三段で説明する構成にした。

## ⚠ 保留・未解決
- `Case` の well-formed 条件をどこまで形式化するかは未確定である。
- `Case` と `Scope` の 1対1 / 1対多 関係をどこまで明示的に定義するかは今後の課題である。
- 実際のケーステンプレート上で、最小構成要素が十分かは `03` の作成後に再点検が必要である。
