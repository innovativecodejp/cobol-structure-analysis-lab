# Research Log: 保証（Guarantee）の数学的形式化

**Date:** 2026-03-02  
**Author:** Cursor Agent  
**Topic:** 保証（Guarantee）の形式的定義と検証可能性の数理的境界

## 1. 背景と目的

移行プロジェクトにおいて「保証」という言葉は、しばしば「既存システムと全く同じ動作をすること」という曖昧な意味で使われる。しかし、この定義は主観的であり、かつ検証不可能である。自動化ツールの適用範囲を明確にし、移行リスクを定量化するためには、保証を数学的な不変条件（Invariant）として再定義する必要がある。

本研究では、Guarantee Unit（L1-L5）の概念に加え、変換関数 $\Phi$ を適用した際の保存条件を形式化し、自動化可能な領域（静的保証）と人間判断が必要な領域（動的保証）の境界を論理的に確定した。

## 2. 決定事項：保証の数理モデル

保証（Guarantee）を、以下の形式で定義することを決定した。

$$
G(GU, \Phi, \mathbb{P}) \iff \forall P \in \mathbb{P}, P(GU) \equiv P(\Phi(GU))
$$

### 2.1 必須となる5つの保存観点 ($\mathbb{P}$)

| 記号 | 名称 | 定義 |
| :--- | :--- | :--- |
| $P_{syn}$ | 構文保存 | 対象言語の文法規則 $\mathcal{L}_{target}$ に適合すること |
| $P_{flow}$ | 制御流保存 | $CFG(S) \cong CFG(\Phi(S))$ （グラフ同型） |
| $P_{data}$ | データ依存保存 | $DFG(S) \subseteq DFG(\Phi(S))$ （定義・使用関係の維持） |
| $P_{side}$ | 副作用保存 | $Trace(IO(S)) \equiv Trace(IO(\Phi(S)))$ |
| $P_{bound}$ | 境界整合性 | $Interface(S) \equiv Interface(\Phi(S))$ |

### 2.2 層別保証モデルの定義

| 層 | 記号 | 定義式 | 検証手段 |
| :--- | :--- | :--- | :--- |
| **構文層** | $G_{syntax}$ | $P_{syn} \land P_{bound}$ | コンパイル |
| **構造層** | $G_{structure}$ | $G_{syntax} \land P_{flow} \land P_{data}$ | 静的解析（CFG/DFG比較） |
| **判断層** | $G_{decision}$ | $G_{structure} \land P_{side}$ | 動的テスト（人間判断含む） |

## 3. 重要な洞察

1.  **静的保証の限界**: $G_{structure}$（構造的保証）まではツールによる数学的証明が可能であるが、$G_{decision}$（判断層保証）は仕様や環境に依存するため、完全自動化は不可能である。この境界を明確に分離した。
2.  **保証崩壊条件**: 動的SQL、動的CALL、ポインタ演算などは、静的解析における決定性（Determinism）を破壊するため、「保証不能（Unknown）」として扱うべきである。
3.  **責任分界点**: 自動変換ツールの責任範囲は $G_{structure}$ の成立までであり、$G_{decision}$ の不一致は「仕様バグ」または「環境差異」として扱われるべきである。

## 5. 結論

「保証」とは、曖昧な安心感ではなく、定義された不変条件 $\mathbb{P}$ が数学的に守られていることである。この定義により、移行プロジェクトにおいて「何を自動化し、何を手動でテストするか」の定量的な配分計画が可能となる。

特に、$G_{structure}$ を満たす領域については、回帰テストの自動生成や形式的検証（Formal Verification）の適用が可能となり、移行コストの大幅な削減が期待できる。
