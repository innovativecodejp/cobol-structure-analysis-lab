# 2026-03-29_03_CasesDescriptionTemplate

## 🎯 今日の目的
- `docs/prompts/phase7/03_Cases-Description-Template.md` に沿って、Case 記述の標準テンプレートを定義する。
- `70_cases` の各ケースを、比較・検証・判断再利用に耐える形で記述できるようにする。

## 🛠 実施内容
- テンプレートの必要性を、記述ばらつきの抑制、比較可能性の確保、Guarantee / Decision 接続の容易化という 3 つの観点から整理した。
- 三層分離（構文層・構造層・判断層）を設計原則として明示した。
- 標準項目として、ID、分類、対象範囲、構造要約、AST / CFG / DFG、Guarantee、判断論点、リスク、分割可能性、検証コストなどを定義した。
- 不明情報、未観測、判定不能の扱いを独立した運用原則として整理した。
- 文書末尾に、そのまま使える Markdown テンプレート本体を掲載した。

## 🔍 結果
- `70_cases` の個別 Case を同一形式で記述するための基本様式ができた。
- 分類、比較、影響分析、検証戦略へそのまま接続できる項目構成になった。
- Case 記述が印象ベースではなく、構造ベースで残せる形になった。

## 💡 学び
- Case 記述では、構造の説明と判断の説明を同じ欄に書かないことが非常に重要だった。
- 判定不能や未観測をテンプレートに含めると、記述の欠落と理論上の保留を区別しやすい。

## 🧠 思考整理
- 良いテンプレートは単なる入力フォームではなく、研究モデルの因果関係を壊さないためのガードレールになる。
- `Case` は比較可能であることが本質なので、最初から比較軸を埋め込んだテンプレートにする必要があった。

## 🧭 抽象レベル
- テンプレートは **文書様式** であると同時に、**構造観測から判断論点へ至る情報配置規則** でもある。

## ⚖ 移行判断への示唆
- Case 記述が揃うと、何が可移行性を阻害しているか、どこで判定不能が生じているかを横断的に見やすくなる。
- 後続の PoC や比較では、このテンプレートがそのまま観測シートとして機能する。
