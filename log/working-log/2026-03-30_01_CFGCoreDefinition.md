# 2026-03-30_01_CFGCoreDefinition

## 🎯 今日の目的
- `docs/30_cfg/01_CFG-Core-Definition.md` に沿い、研究空間における **CFG の中核定義** を作業ログに要約する。
- AST / IR の後に CFG が要る理由と、判断接続層（Guarantee / Scope / Decision）への **構造入力** としての役割を一文脈で押さえる。

## 🛠 実施内容
- CFG を **「制御到達と経路閉包の構造層」** と一文定義し、実行トレース図でも AST の別表現でもないことを明文化した内容を整理した。
- AST 単体の不足と、IR 単体では **グラフとしての遷移骨格が固定されない** 点を対比した。
- CFG の責務（扱う／扱わない）と、最小語彙（ノード・辺・経路・領域）の位置づけを要約した。
- COBOL の **制御解析単位** と **業務参照単位（paragraph / section）** の二層性を確認した。
- 他モデル（DFG、Guarantee / Scope / Decision）との関係表・接続の前提を読み取った。

## 🔍 結果
- Phase9 `30_cfg` の出発点として、「CFG は経路と閉包を判断可能にする外在化」という **研究上の制御抽象** が言語化された。
- 定義不全時のリスク（偽直列化、PERFORM/GO TO の説明不能、DFG/Scope との不整合）が列挙され、後続章の動機が明確になった。

## 💡 学び
- CFG は「制御フロー図の一般像」とは目的が違い、**分岐・合流・経路集合の閉じ方** を COBOL 実効構造に即して保持する層だと理解した。
- 層の積み上げは AST→IR→CFG/DFG→判断層で、CFG は **到達・経路の操作可能な表現** に特化している。

## 🧠 思考整理
- 移行判断で「どの制御構造に対して主張するか」を固定するには、IR の型付けだけでなく **ノード・辺としての遷移順序** が必要だと整理した。
- 未解決論点（閉包の精密化は `08`、動的制御の静的埋め込みは方針保留）は、本稿では **境界を切って先送り** されている。

## 🧭 抽象レベル
- CFG は **制御到達関係の骨格** と **経路・閉包の前提** を与える **中間構造層**。

## ⚖ 移行判断への示唆
- Guarantee / Scope / Decision は、CFG なしでは **経路依存性・境界候補・リスクパターン** の根拠を一貫して説明しにくい。
- paragraph 名だけでは閉じない解析単位を前提に、説明責任用の **トレーサビリティ** を失わない設計が必要。
