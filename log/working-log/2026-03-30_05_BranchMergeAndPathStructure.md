# 2026-03-30_05_BranchMergeAndPathStructure

## 🎯 今日の目的
- `docs/30_cfg/05_Branch-Merge-and-Path-Structure.md` に沿い、CFG を **経路の構造** として読むための branch・merge・path を要約する。
- merge を **判断上の再統合点** とし、Guarantee の **経路依存性** の根拠を作業ログに残す。

## 🛠 実施内容
- branch（二つ以上の後続から一つを選ぶ）と、分岐の開き方（各選択肢が **別辺・別経路接頭辞**）を整理した。
- merge（複数前駆から同一ノードへ）を **再統合点** かつ **path-sensitive な観測点** として位置づけた。
- path を **辺の有限列** と定義し、ノード集合の一致が経路の一致ではないことを強調した内容を要約した。
- 構文的経路／実行可能経路／保証対象経路／未カバー経路という **準備概念** を整理した。
- 多分岐（EVALUATE 等）の扇状 branch と共通 merge、merge 省略時の **並行終端** と保証分割の含意を確認した。
- EXIT / GO TO 等の **早期脱出** が merge を経由しない経路を生じ、path 集合を分断しうる点を整理した。
- Guarantee / Scope / Decision への接続（準備段階）を要約した。

## 🔍 結果
- 静的 CFG は構文的経路の上限近似であり、**feasible 経路** や **保証対象** は別段階で絞り込む、という枠組みが明確になった。
- merge 欠落＋終端増加が **経路集合の指数的肥大** とコスト増につながりうる、というリスク観が言語化された。

## 💡 学び
- Guarantee が状態不変等を含むとき、**どの分岐選択列を通ったか** が前提になり、保証単位は経路依存になりうる。
- 早期脱出は必ずしも non-structured ではないが、**合流の欠落** と併発しうる。
