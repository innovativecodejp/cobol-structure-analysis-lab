# 2026-03-29_01_CasesCoreDefinition

## 🎯 今日の目的
- `docs/prompts/phase7/01_Cases-Core-Definition.md` に沿って、`70_cases` の基礎概念として `Case` を定義する。
- `Case` を単なる事例ではなく、構造観測・保証要求・移行判断を束ねる単位として整理する。

## 🛠 実施内容
- プロンプトで要求されている章立て（目的、必要性、定義、最小構成要素、Scope / Guarantee / Decision との関係、抽象度、判断検証単位としての理由、全体への意味、まとめ）を確認した。
- `60_scope` の基礎定義文書を参照し、`Case` も同様に研究上の第一級概念として書く方針を取った。
- `Case` と AST ノード、Guarantee Unit、Decision 結論を混同しないように、差異を個別に言語化した。
- 後続の `02`〜`10` へ接続しやすいよう、三層（構文・構造・判断）の整理を入れた。
- `log/research-log/2026-03-29_01_CasesCoreDefinition.md` に研究観点を記録した。

## 🔍 結果
- `Case` を「事例」ではなく **構造的判断単位** として固定できた。
- 後続の分類・テンプレート・比較・検証が、何を対象にしているかを定義できた。
- `Scope / Guarantee / Decision` との接続が明確になり、`70_cases` を独立した理論フェーズとして始められる形になった。

## 💡 学び
- `Case` は対象領域そのものではなく、対象領域に観測・保証・判断を載せる **研究記述の単位** として書くと整理しやすい。
- `Case` を構文単位に寄せすぎると AST の再説明になり、判断単位に寄せすぎると Decision の再説明になるため、中間層としての位置取りが重要だった。

## 🧠 思考整理
- `70_cases` は理論の適用層であると同時に、理論を壊す反例を見つける層でもある。
- そのため `Case` は、比較可能性と反証可能性を最初から含んだ定義でなければならない。

## 🧭 抽象レベル
- Case / Scope / Guarantee / Decision の役割差：**対象領域**、**保存主張**、**判断結論**、**判断検証単位** を分けて扱う。

## ⚖ 移行判断への示唆
- `Case` が明確であるほど、移行可否や分割要否の判断理由を構造的に説明しやすくなる。
- `Case` が曖昧なままだと、保証要求と判断対象がずれ、検証証拠の範囲も不安定になる。
