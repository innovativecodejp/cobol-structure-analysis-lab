# 02_Cases-Taxonomy

## 目的
`70_cases` における **ケース分類体系（Taxonomy）** を設計してください。  
この文書の目的は、ケースを無秩序に集めるのではなく、**構造的差異に基づく代表パターンとして整理できる分類軸**を確立することです。

---

## この文書で扱う範囲
扱う内容：

- Case を分類する意義
- 分類軸の設計
- 分類粒度の設計
- 代表ケースの選定基準
- 分類体系と判断再利用性の関係

扱わないこと：

- 各ケースの詳細な中身
- 個別案件ベースの業務説明
- 実データに基づく統計分類
- スコアリングの数式化

---

## 前提
Case は、単なるサンプルではなく、**構造的判断単位**です。  
したがって分類体系も、表面的な機能別分類ではなく、以下に関わる構造軸を中心に設計してください。

- 制御構造
- データ構造
- I/O 構造
- 外部依存
- 保証困難性
- 影響伝播性
- 分割移行可能性
- 境界明瞭性

---

## 必須観点
必ず以下を含めてください。

1. **Taxonomy の目的**
   - なぜ分類が必要か
   - ケース数を増やすことより分類品質が重要である理由

2. **主要分類軸**
   - 制御構造中心ケース
   - データ構造中心ケース
   - I/O 中心ケース
   - 外部依存中心ケース
   - 保証困難ケース
   - 影響伝播が大きいケース
   - 境界曖昧ケース
   - 分割移行しやすいケース / しにくいケース

3. **分類粒度**
   - 単一構文ケース
   - 複合構造ケース
   - 業務責務ケース
   - 連携・システム境界ケース

4. **分類の設計原則**
   - 相互排他性と重複許容の扱い
   - 主分類と副分類の考え方
   - 1 Case が複数の軸をまたぐ場合の扱い

5. **分類と判断モデルの関係**
   - 分類が Guarantee/Decision の再利用性にどう寄与するか
   - 代表ケース選定にどう使うか

6. **今後の利用方法**
   - ケース収集方針
   - ケース比較方針
   - ケースカバレッジの考え方

## 要求品質
- 単なるカテゴリ一覧ではなく、研究上の分類原理として記述すること
- 分類軸同士の意味の違いを明確にすること
- 「構文分類」だけに偏らず、構造・判断との接続を含めること
- `70_cases` 全体の運用指針になる内容にすること
