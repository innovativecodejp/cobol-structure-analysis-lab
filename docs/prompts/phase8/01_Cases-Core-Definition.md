# 01_Cases-Core-Definition

## 目的
`70_cases` における **Case の定義** を確立してください。  
この文書では、Case を単なる事例やサンプルではなく、**移行判断を検証するための構造的単位**として定義することを目的とします。

---

## この文書で扱う範囲
扱う内容は以下に限定してください。

- Case の研究上の役割
- Case の定義
- Case の最小構成要素
- Case と Scope / Guarantee / Decision の関係
- Case がなぜ判断検証単位になるのか
- 今後の `70_cases` 全体に対する基礎定義

扱わないこと：

- 個別ケースの詳細記述
- 具体的な COBOL サンプルの列挙
- 実装戦略やコード変換手順
- 評価スコアリングの詳細設計

---

## 前提
本研究は、以下の流れの上にあります。

- `10_ast`：構造観測の基盤
- `50_guarantee`：保証単位・保証空間
- `60_decision`：移行判断軸
- `70_cases`：ケースによる適用・検証層

したがって、この文書では Case を、上記の理論を現実の判断材料へ接続するための単位として位置づけてください。

---

## 必須観点
必ず以下を含めてください。

1. **Case の定義**
   - 事例ではなく構造的判断単位であること
   - 何をもって 1 Case とみなすか
   - 構文・構造・判断の3層との関係

2. **Case の必要性**
   - なぜ AST / Guarantee / Decision だけでは不十分なのか
   - なぜ Case 層が必要なのか

3. **Case の構成**
   - 最小観測単位
   - Case を記述するために必要な要素
   - Scope と異なる点
   - Guarantee Unit と異なる点
   - Decision 対象と異なる点

4. **Case の役割**
   - 構造モデルの適用対象
   - 保証割当の検証対象
   - 判断モデルの説明対象
   - 影響分析の観測対象

5. **Case の抽象度**
   - 構文層
   - 構造層
   - 判断層
   の3レベルを分けて説明すること

6. **研究上の到達点**
   - この定義が `70_cases` 全体でどのような基盤になるか
   - Case 定義が曖昧だと何が破綻するか
