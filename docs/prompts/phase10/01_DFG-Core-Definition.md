# Phase10 / 40_dfg
## 01_DFG-Core-Definition

あなたは「COBOL構造解析研究室」の研究補助AIです。  
本タスクの目的は、COBOL資産解析における **DFG（Data Flow Graph）の中核定義** を確立することです。

---

## 1. このタスクの位置づけ

本研究は以下の蓄積の上にあります。

- 10_ast: 構文構造の基底
- 20_ir: 中間表現による意味抽象
- 30_cfg: 制御遷移構造の形式化
- 50_guarantee: 保証単位と保証空間
- 60_scope: 影響境界・閉包・検証範囲
- 70_cases: 具体事例による照合

この 40_dfg では、**値がどこで定義され、どこへ伝播し、どこで使用されるか** を構造的に表現する。  
ここで扱うのは最適化器向けの低レベルデータフローではなく、**移行判断・影響分析・保証可能性評価のための構造的DFG** である。

---

## 2. 作成対象

以下を完全な形で記述してください。

1. DFG の研究目的
2. DFG の定義
3. DFG が表現すべき対象
4. DFG の最小構成要素
5. DFG と AST / IR / CFG / Guarantee / Scope の関係
6. DFG が移行判断に与える価値
7. DFG の抽象度（構文層 / 構造層 / 判断層）

---

## 3. 必須で含める内容

### 3.1 DFG の定義
以下の観点を明示してください。

- ノードとは何か
- エッジとは何か
- 「定義」「使用」「再定義」「伝播」「集約」「分岐合流」の扱い
- 値の同一性と変数名の同一性を分離すべきか
- 文単位・段落単位・手続き単位のどこまでを1つの流れとして扱うか

### 3.2 DFG の役割
以下を整理してください。

- 変更影響分析
- 未保証領域の検出
- データ依存に基づく移行単位の補助
- CFG では見えない依存の可視化
- Scope 閉包の根拠生成
- Guarantee Unit の前提/結果関係の補助

### 3.3 抽象度
次の3層を分けて記述してください。

- 構文層: MOVE / COMPUTE / ADD / READ / WRITE 等の構文的出現
- 構造層: define-use / use-define / data dependency / alias-like relation
- 判断層: 「この項目変更はどこまで波及するか」「どこまで保証対象に含めるべきか」
