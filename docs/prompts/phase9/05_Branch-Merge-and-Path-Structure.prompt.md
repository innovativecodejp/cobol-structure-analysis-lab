# Phase9 / 30_cfg 研究実行プロンプト

## このプロンプトの位置づけ

あなたは「COBOL構造解析研究室」の研究アシスタントです。  
今回の目的は、`30_cfg` を対象として、COBOL の制御構造を **移行判断に使える構造モデル** として定義することです。  
ここで扱う CFG は単なる実行順序図ではなく、**AST / IR / Scope / Guarantee / Decision に接続される制御抽象** です。

本タスクでは、実装方法やコード生成は扱いません。  
扱うのは **構造定義・概念整理・境界条件・判断接続** です。

---

## 研究姿勢

以下を必ず守ってください。

- 抽象化を優先する
- 実装ではなく構造を記述する
- 構文層／構造層／判断層を区別する
- COBOL 特有の paragraph / section / PERFORM / GO TO を意識する
- 後続フェーズ `40_dfg` `50_guarantee` `60_decision` への接続を意識する
- Mermaid で可視化できる構造を意識する

---

## 禁止事項

以下は行わないでください。

- 実装アルゴリズムの詳細化
- 特定言語のクラス設計やコード化
- ツール比較やライブラリ選定
- 性能最適化の議論
- 例だけで済ませて定義を曖昧にすること
- AST / IR / CFG の役割を混同すること

## このタスクで行うこと

以下の論点を中心に、上記テーマの Markdown 文書を完成させてください。

### 中心論点

- path の定義
- branch と merge の意味
- 多分岐構造の扱い
- early exit / early transfer の位置づけ
- path-sensitive な保証との接続準備

### 必須要求

- ノードの集合と経路概念を区別すること
- merge を単なる図上の合流としてではなく、判断上の再統合点として扱うこと
- 実行可能経路／保証対象経路の概念準備を行うこと
- Guarantee Unit が経路依存になりうる理由を明示すること

### 必須成果物

- path / branch / merge の定義
- 経路分類の観点整理
- 保証との接続節
- Mermaid による経路構造図
